# 6. Planning Rules Specific to This Thesis

## 6.1 Proxy-state language

Clinical tags must normally be described as:

* derived proxy states;
* proxy phenotypes;
* clinically inspired proxy labels;
* weak clinical labels.

Do not call them verified diagnoses.

Do not call them formal latent variables without an explicit theoretical and validation argument.

When discussing the repository’s historical naming, distinguish the code’s `latent` terminology from the preferred thesis terminology.

## 6.2 Prediction versus causal analysis

Keep separate:

1. predicting proxy-state labels from irregular measurements;
2. aggregating or normalizing predicted labels;
3. treating selected proxy states as exposures in downstream analyses;
4. estimating adjusted associations or causal effects under assumptions.

Predictive AUROC or AUPRC does not validate causal identification.

Mortality-prediction performance does not validate a proxy state as a diagnosis.

## 6.3 Causal wording

The thesis may describe:

* implemented causal estimators;
* DAG-guided adjustment logic;
* estimates produced under stated assumptions;
* sensitivity and overlap diagnostics.

It must not imply unconditional causality.

Any planned causal conclusion must explicitly address:

* consistency;
* exchangeability or no unmeasured confounding;
* positivity/overlap;
* model specification;
* measurement error;
* treatment definition;
* temporal ordering;
* proxy-label error;
* intervention interpretability.

## 6.4 Active versus legacy code

Treat files under:

```text
causal-irregular-time-series/src/draft/
```

as legacy unless execution evidence proves they were used for final results.

Do not design the main methods chapter around a legacy implementation merely because its code is easier to explain.

If active and legacy implementations conflict, record the conflict rather than silently choosing one.

## 6.5 Dataset contracts

Account for the fact that the causal repository and STraTS repository use different processed-pickle contracts.

The plan must make this distinction explicit and identify where the router or preprocessing workflow bridges them.

Do not describe them as one identical artifact contract.

## 6.6 Missing result artifacts

The Stage 2 audit identified that many outputs may be ignored, external, archived, or absent.

A section can be planned without results, but its result claims and tables must remain blocked.

Do not transform implementation evidence into execution evidence.

## 6.7 Example thesis use

Use `example-omri-thesis` to understand practical LaTeX patterns.

Do not assume its package list is clean or ideal.

Document duplicate, obsolete, fragile, or project-specific LaTeX choices that should not automatically be copied.

Examples may include:

* repeated package imports;
* mixed bibliography tooling;

# 7. Validation Procedure

After creating the planning files, perform the following checks.

## 7.1 File and link validation

Check that all repository paths cited in planning files exist.

Where a path is expected but absent, mark it explicitly.

## 7.2 Citation-key validation

For every citation key mentioned in planning files:

* confirm it exists in `metadata/references.bib`;
* confirm its corpus role in `catalog.csv`;
* flag missing or inconsistent keys.

Do not modify bibliography files.

## 7.3 Evidence-status validation

Confirm that no planned claim has been promoted beyond its evidence level.

Examples:

* code existence → implementation-confirmed only;
* output file without provenance → provenance unclear;
* verified summary CSV linked to a run → result-confirmed, if provenance is sufficient;
* literature statement → background support, not repository execution proof.

## 7.4 BGU requirement validation

Cross-check the proposed thesis sequence and planned sections against the official PDF.

Do not rely exclusively on the example thesis.

## 7.5 Scope validation

Confirm that Stage 3 did not:

* draft full thesis prose;
* modify code;
* modify result files;
* modify literature files;
* modify the example thesis;
* create final LaTeX chapters;
* generate figures;
* generate numeric result tables;
* commit or push.

## 7.6 Optional lightweight checks

You may run lightweight, read-only or planning-validation commands such as:

```bash
git status --short
find thesis-writing/planning -maxdepth 1 -type f -print
rg "\[NEEDS|\[ADVISOR|\[PROVENANCE|\[VALIDATION" thesis-writing/planning
```

Do not run model training, preprocessing, causal estimation, or other expensive workflows.

Do not compile or edit the example thesis merely to prove that it builds.

A LaTeX compile test belongs in a later construction or validation stage unless an existing CliniCause thesis skeleton already exists and can be checked without modifying it.

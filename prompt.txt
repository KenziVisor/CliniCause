We are debugging the thesis repository and specifically the MIMIC preprocessing stage launched through router.py.

Problem:
router.py runs src/preprocess_mimic_iii_large.py for MIMIC preprocessing. Under a 29G Slurm memory limit, the preprocessing subprocess is killed with return code -9/SIGKILL. This is likely OOM, not a Python exception.

Important context:
- Git history shows that chunksize=10000000 was already present from the initial MIMIC preprocessing implementation, so there may not be an older smaller chunksize to restore.
- The real problem is not only the large chunksize. The current script reads CHARTEVENTS in chunks but appends all filtered chunks to `ch`, then does `pd.concat(ch)`, so it still materializes a huge filtered CHARTEVENTS dataframe.
- LABEVENTS and other large MIMIC tables may also be materialized too aggressively.
- The code repeatedly does `events = pd.concat([events, new_df])` on a growing dataframe, which increases peak memory.
- We want preprocessing to actually run under 29G, not skip preprocessing or reuse an existing artifact.

4. Refactor away from full raw-table materialization
   The current pattern:

       ch = []
       for chunk in read_csv(..., chunksize=...):
           ...
           ch.append(chunk)
       ch = pd.concat(ch)

   must be removed.

   Instead:
   - Read raw tables in chunks.
   - Filter each chunk early.
   - Write filtered/intermediate shards to disk under --tmp-dir.
   - Never keep the full filtered CHARTEVENTS/LABEVENTS/INPUTEVENTS/OUTPUTEVENTS dataframe in memory.

5. Early filtering rules
   For each raw table, keep only needed columns and filter as early as possible.

   CHARTEVENTS:
   - use columns: HADM_ID, ICUSTAY_ID, ITEMID, CHARTTIME, VALUE, VALUENUM, VALUEUOM, ERROR
   - keep rows where ICUSTAY_ID is in adult ICU stays
   - ERROR != 1
   - CHARTTIME not null
   - not both VALUE and VALUENUM null
   - ITEMID is in the union of all CHARTEVENTS item IDs used by the feature extraction logic
   - add TABLE="chart"

   LABEVENTS:
   - read in chunks too
   - use columns: HADM_ID, ITEMID, CHARTTIME, VALUE, VALUENUM, VALUEUOM
   - keep HADM_ID in retained admissions
   - CHARTTIME not null
   - not both VALUE and VALUENUM null
   - ITEMID is in the union of all LABEVENTS item IDs used by feature extraction
   - add ICUSTAY_ID=np.nan
   - add TABLE="lab"

   INPUTEVENTS_CV / INPUTEVENTS_MV / OUTPUTEVENTS:
   - apply equivalent early filtering using the current code’s columns and item IDs
   - keep only the columns required downstream
   - normalize columns to the event schema as early as possible when safe:
     HADM_ID, ICUSTAY_ID, CHARTTIME, ITEMID, VALUE, VALUENUM, VALUEUOM, TABLE

6. Feature extraction structure

7. Global median/statistics pass
   For every feature where the current code uses a median to replace invalid/outlier values:
   - compute the median globally across all relevant filtered shards
   - do not compute median per chunk
   - store medians in a dictionary
   - log feature name, valid count, and median value

   This may require a two-pass flow:
   - pass A: collect valid values for median computation from shards
   - pass B: transform shards using the global medians

   Use memory-safe accumulation. If a feature has many values, do not keep all raw values if avoidable; but correctness is more important. If exact median requires collecting values, collect only that feature’s numeric values, not entire raw rows. If exact median is still too large, write values to temporary shard files and compute median from those.

8. Transformation pass

   Avoid `events = pd.concat([events, ...])` on a growing dataframe.
   Prefer writing transformed event shards to disk.

9. ICU stay assignment
   The current code assigns missing ICUSTAY_ID by HADM_ID and CHARTTIME:
   for each event with missing ICUSTAY_ID, assign the ICU stay from the same HADM_ID where:

       INTIME <= CHARTTIME <= OUTTIME

   Preserve this logic.

   But do it shard-by-shard:
   - Build a small ICU interval lookup from the ICU table.
   - For each transformed event shard:
     - assign missing ICUSTAY_ID using the same rule
     - drop rows that cannot be assigned
     - write aligned shard back to disk

   Avoid merging the entire events dataframe with all ICU intervals at once.

10. Timeline conversion and cohort filtering
   Preserve the existing cohort logic:
   - ICU stay lasted at least 24h
   - patient alive for at least 24h from INTIME, or DEATHTIME is null
   - ICU stay has at least one event in the first 24h
   - keep only events belonging to final retained ICU stays

   Implement this in a memory-safe way:
   - Compute rel_charttime/minute per shard by joining to a small ICUSTAY_ID -> INTIME mapping.
   - Determine final eligible ICU stays.
   - Filter shards by final eligible ICU stays.
   - Add Age and Gender static rows at minute=0 exactly as the current code does.

11. Final aggregation
   The current final canonicalization uses groupby over:

       ["ts_id", "minute", "variable"]

   and computes mean value.

   For memory safety, use map-reduce:
   - Per shard:
     - rename columns to ts_id/minute/variable/value
     - group by ["ts_id", "minute", "variable"]
     - compute sum and count
     - write aggregate shard
   - Final:
     - concatenate aggregate shards
     - group by ["ts_id", "minute", "variable"]
     - sum the sums and counts
     - compute value = total_sum / total_count

   This preserves exact mean semantics and avoids mean-of-means.

   Then build:
   - ts
   - ts_ids
   - oc

   using preprocess_mimic_iii_large_contract.py where possible.

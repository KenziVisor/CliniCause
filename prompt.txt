# Stage 2.1 — Read-Only Repository Structure Audit

## Context

You are operating at the root of the `CliniCause` Git repository.

This is the first task in Stage 2 of the thesis-writing workflow: Repository and Evidence Audit.

The purpose of this task is to understand the repository structure before examining individual pipelines or writing thesis prose.

Treat the repository contents as evidence. Do not assume that a script, notebook, configuration, checkpoint, result, figure, or table was actually used merely because it exists.

## Objective

Inspect the complete repository and create a factual repository map that identifies the important directories, source files, notebooks, configurations, experiments, outputs, figures, tables, models, thesis resources, and potentially obsolete or duplicated material.

This task is structural only. Do not yet perform a detailed audit of model methodology, data processing, clinical definitions, causal assumptions, or numerical results.

## Allowed Scope

You may inspect:

* the complete `CliniCause` repository;
* directory and file names;
* small text-based source files when needed to determine their purpose;
* Git-tracked file information;
* `.gitignore`;
* configuration and environment files;
* README and documentation files;
* Git history when useful for identifying obsolete or superseded material.

You may create or edit only:

```text
thesis-writing/audit/repository_map.md
thesis-writing/audit/unresolved_questions.md
```

Create the `thesis-writing/audit/` directory if it does not exist.

## Forbidden Scope

Do not:

* modify source code;
* modify notebooks;
* modify configurations;
* modify data;
* modify model checkpoints;
* modify experiment outputs;
* modify figures or tables;
* modify the literature corpus;
* modify thesis prose or LaTeX;
* install packages;
* download files;
* execute training, preprocessing, evaluation, or analysis pipelines;
* rerun notebooks;
* regenerate results;
* delete, move, rename, or reorganize repository files;
* commit or push changes.

Do not recursively read large datasets, model checkpoints, binary files, cache directories, generated environments, or other large artifacts merely to map the repository.

## Required Procedure

### 1. Confirm the workspace

Confirm that the Git repository root is `CliniCause`.

Record:

* repository root;
* current branch;
* whether the working tree already contains changes.

Do not alter existing changes.

### 2. Inspect the top-level structure

Identify all important top-level directories and files.

For each relevant entry, determine its apparent purpose using repository evidence rather than its name alone.

### 3. Classify repository contents

Classify relevant directories and files into categories such as:

* data access or raw-data references;
* preprocessing and cohort construction;
* time-series construction or representation;
* predictive-model implementations;
* training and evaluation;
* clinical proxy-state construction;
* causal-inference analysis;
* overlap diagnostics;
* sensitivity analysis;
* notebooks;
* configurations;
* experiment definitions;
* stored outputs and results;
* model checkpoints;
* figures;
* tables;
* documentation;
* tests;
* utilities;
* thesis-writing resources;
* archived, duplicated, generated, or potentially obsolete material.

Use additional categories where the repository requires them.

### 4. Identify important entry points

Identify apparent entry points such as:

* main scripts;
* command-line interfaces;
* orchestration scripts;
* central notebooks;
* configuration roots;
* result-generation scripts;
* environment files;
* Makefiles or task runners.

Do not claim that an entry point was executed unless execution evidence exists.

### 5. Inspect Git context where useful

Use Git history selectively when it helps determine whether:

* a directory is current or obsolete;
* similarly named files represent different versions;
* an output appears generated by a known workflow;
* files were recently replaced or superseded.

Clearly distinguish Git-supported conclusions from tentative interpretations.

### 6. Record uncertainty

Whenever a directory or file purpose cannot be determined confidently, record it as unresolved rather than guessing.

Add such items to:

```text
thesis-writing/audit/unresolved_questions.md
```

You are working on the GitHub repository **CliniCause**, a unified repository for the thesis causal pipeline and the STraTS fork.

Goal: perform a high-level health check that the codebase is not broken, **without assuming access to the real datasets** and without running the full pipeline end-to-end.

Important constraint:
You cannot validate scientific correctness or full runtime correctness because the data is unavailable. Your job is to verify that the repository is structurally coherent, importable, configurable, and that every pipeline stage has a minimal “does not immediately break” check.

Work in a loop:

1. Inspect the repository structure and identify the intended pipeline stages.
2. For each stage, define the weakest meaningful check that can run without real data.
3. Run the check.
4. If it fails, fix the cause.
5. After fixing, rerun the failed check.
6. Then rerun all earlier checks that could have been affected.
7. Continue until all checks pass.
8. At the end, report exactly what was checked, what was fixed, what was not possible to check without data, and what assumptions remain.

Expected pipeline order:

1. Preprocessing
2. Latent-variable tagging
3. Decision-tree plot / visualization
4. STraTS representation / model stage
5. Thesis main causal pipeline

Start by mapping the actual files/scripts to these stages. Do not invent new architecture unless necessary. Prefer small fixes over refactors.

Minimum checks to perform:

A. Repository structure check

* Verify that the expected top-level entry points exist.
* Locate `router.py`, thesis-side scripts, STraTS-side scripts, configs, README/SCRIPTS docs if present.
* Verify that paths referenced by `router.py` actually exist.
* Verify that shell scripts referenced by the router exist and are executable or at least syntactically valid.

B. Python import / syntax check

* Run Python compile checks over the repository, excluding data, outputs, runs, checkpoints, virtual environments, caches, and large generated folders.
* Try importing key modules where safe.
* Do not import modules that immediately require real data at import time; if that happens, fix the module so data access happens inside `main()` / runtime functions, not during import.

Suggested checks:

* `python -m compileall ...`
* targeted imports for router and core modules
* check that each script has a usable `main` guard where appropriate

C. Router check

* Inspect `router.py`.
* Verify that CLI flags are parsed correctly.
* Verify that each stage can be selected independently.
* Verify that dataset-related flags are passed consistently.
* Verify that paths passed from the router match the actual script interfaces.
* Verify that dry-run behavior exists or add a minimal dry-run mode if missing.
* Dry-run mode should print the resolved commands/stages without requiring data.

D. Config check

* Validate that config files exist.
* Validate that CSV/config readers use existing column names.
* Validate that required keys are documented or checked explicitly.
* If configs contain paths to data, the code should fail gracefully with a clear message, not with a cryptic stack trace.
* Add validation helpers if needed, but keep them small.

E. Preprocessing stage check

* Do not require real PhysioNet or MIMIC data.
* Verify that the preprocessing script can show help / parse args.
* Verify that required input paths are validated before heavy logic starts.
* If possible, add a `--dry-run` or `--validate-only` path that checks arguments and prints planned outputs.
* The check passes if the script can be invoked in dry-run/validate mode and fails gracefully when data is missing.

F. Latent-variable tagging check

* Verify both PhysioNet and MIMIC tagging entry points.
* Pay attention to inconsistent argument naming, e.g. `--pkl-path` vs `--pkl_path`, and make sure the router calls the correct interface.
* Verify that output paths are constructed consistently.
* Verify that decision-tree/rule dictionaries are created only after input validation.
* The check passes if each tagging script can parse args and perform dry-run/validation without real data.

G. Decision-tree plot / visualization check

* Verify that the plotting script can locate its expected inputs through arguments/config.
* It should fail gracefully if the required latent-tag outputs are missing.
* Check that output directories are created safely.
* Avoid requiring actual plots to be generated without input data.
* The check passes if CLI/config validation works and missing-input errors are clear.

H. STraTS stage check

* Inspect STraTS shell scripts and Python entry points.
* Specifically check bash syntax with something like `bash -n`.
* Fix any broken `if/fi`, quoting, path, or argument-passing issues.
* Verify that STraTS scripts do not assume they are always launched from a specific working directory unless documented or enforced.
* Verify that the router passes the minimal required variables into STraTS.
* The check passes if shell syntax is valid and dry-run/argument validation works without data.

I. Thesis main causal pipeline check

* Verify that the main thesis entry point is invoked the intended way, preferably through existing scripts such as `scripts/run_main.sh` if that is the current contract.
* Verify that model/dataset/downsample/config flags are wired correctly.
* Verify that missing-data errors are explicit and early.
* Do not try to run the actual estimators without data.
* The check passes if the main script can parse/validate configuration and fail gracefully before data-dependent execution.

J. Shell script check

* Run syntax checks on all relevant `.sh` files:

  * `bash -n script.sh`
* Check that scripts use safe path handling.
* Check that scripts do not silently continue after critical failures.
* Prefer `set -euo pipefail` where safe, but avoid breaking scripts that intentionally allow optional commands.

K. Documentation consistency check

* Compare README/SCRIPTS examples against actual commands.
* Fix outdated commands only if clearly wrong.
* Make sure the documented pipeline order matches the router.
* Add a short “health check without data” section if missing.

Required loop behavior:
For every failed check:

1. Identify the smallest root cause.
2. Patch it.
3. Rerun the failed check.
4. Rerun any earlier checks that may be affected.
5. Continue.

Do not stop after the first fix. The task is complete only when the high-level no-data health-check loop passes.

Do not:

* Download datasets.
* Add fake clinical data unless a tiny synthetic smoke test already fits naturally and safely.
* Refactor the whole repository.
* Change scientific logic unless required to prevent obvious breakage.
* Hide failures by catching all exceptions.
* Make the router silently skip stages unless explicitly requested through flags.
* Claim the full pipeline works without data.
